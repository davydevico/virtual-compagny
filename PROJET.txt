SURGIFLOW COMPANY OS — Description du projet
=============================================

SURGIFLOW est une plateforme SaaS B2B de suivi péri-opératoire (J-15 à J+15)
qui automatise la communication chirurgien-patient via SMS/WhatsApp.

Ce projet est l'OS interne de SURGIFLOW : une application web permettant
au CEO (Davy) de gérer une équipe de 13 agents IA virtuels spécialisés,
chacun ayant une expertise métier, une mémoire persistante, et la capacité
de collaborer et déléguer de façon totalement autonome.


STACK TECHNIQUE
---------------
- Frontend  : Next.js 14 + TypeScript + Tailwind CSS (thème sombre)
- Base de données : Supabase (PostgreSQL)
- LLM : Claude API (claude-sonnet-4-20250514)
- Déploiement : Netlify (avec @netlify/plugin-nextjs)


BASE DE DONNÉES (5 tables Supabase)
------------------------------------
- agents : profil de chaque agent (nom, rôle, département, avatar, prompt système, manager)
- memories : historique de tous les échanges CEO ↔ agent (mémoire persistante)
- recruitment_requests : demandes de recrutement générées par les C-Levels
- projects : projets lancés par le CEO avec leur statut
- project_logs : logs en temps réel de l'exécution multi-agents

PHILOSOPHIE DU SYSTÈME DE DÉLÉGATION
--------------------------------------
Le CEO parle aux managers (Marcus, Rachel, Sophie, etc.).
Les managers délèguent automatiquement à leur équipe.
Les agents travaillent et livrent des livrables concrets.
Le manager synthétise et rapporte au CEO.
Le CEO n'a rien d'autre à faire que valider.

Les agents NE PRÉTENDENT PAS avoir exécuté du code ou poussé sur GitHub.
Ils livrent des plans d'action, du code prêt à copier-coller, des configs
et des instructions ultra-détaillées que le CEO peut intégrer lui-même.


FONCTIONNALITÉS
---------------

1. DASHBOARD (app/dashboard/page.tsx)
   - Vue organigramme de toute l'équipe groupée par département
   - Statut de chaque agent (actif, en mission)
   - Stats globales (nombre d'agents, projets en cours, projets terminés)
   - Alerte si des demandes de recrutement sont en attente
   - Skeleton loading animé pendant le chargement

2. RECRUTEMENT (app/recruit/page.tsx)
   - Le CEO recrute directement les 5 C-Levels (Marcus, Rachel, Sophie, Inès, David)
   - À chaque recrutement, Claude génère automatiquement 2-4 demandes de recrutement
     pour les subordonnés de ce C-Level
   - Les demandes arrivent dans un panneau de validation
   - CEO approuve ou refuse chaque demande
   - À l'approbation, l'agent est créé en DB et devient actif
   - Matching recrutement par nom (pas par rôle) pour éviter les faux négatifs

3. CHAT 1-TO-1 avec DÉLÉGATION AUTONOME (app/chat/[agentId]/page.tsx)

   Flow complet quand le CEO parle à un manager (ex : Marcus) :

   ÉTAPE 1 — Réponse initiale du manager (2-5 sec)
     → POST /api/chat : Marcus répond, retourne sa réponse + liste de délégations JSON
     → La réponse s'affiche immédiatement dans le chat

   ÉTAPE 2 — Cartes de délégation apparaissent dans le fil
     → Pour chaque agent délégué (Samy, Léa, Omar, Nina...) une carte s'insère
       dans le chat avec le statut "En attente" (point gris)
     → Les cartes montrent le brief exact envoyé par le manager

   ÉTAPE 3 — Brief des agents (par paquets de 2 pour éviter timeout Netlify)
     → POST /api/brief pour chaque agent
     → La carte passe en "En cours..." (point orange pulsant + dots animés)
     → Quand l'agent répond, carte passe en "Livré ✓" (point vert)
     → Retry automatique (1 fois) si timeout ou erreur réseau

   ÉTAPE 4 — Synthèse du manager
     → POST /api/synthesize : le manager reçoit tous les livrables et produit
       un guide d'implémentation complet pour le CEO
     → Format imposé : 🎯 Mission / 📋 Étapes détaillées / ⚡ Ordre d'exécution / ✔️ Vérification

   INDICATEURS VISUELS :
   - Avatar du manager avec anneau violet pulsant + "orchestre l'équipe..." pendant le process
   - Chaque agent apparaît comme une ligne séparée dans le fil de chat
   - Point de couleur (gris/orange/vert/rouge) mis à jour en temps réel
   - Bouton "Voir le détail ↓" pour lire le livrable complet de chaque agent
   - Lien direct vers le chat de chaque agent délégué

   PERSISTANCE DE SESSION :
   - Les cartes de délégation sont sauvegardées en localStorage (TTL 30 min)
   - Si l'utilisateur recharge la page, la session est restaurée
   - Les agents "En cours" au moment du refresh passent en "Erreur" (honnêteté)
   - Avertissement navigateur (beforeunload) si rafraîchissement pendant le process
   - Session effacée proprement dès que la synthèse est reçue

   MÉMOIRE PROPRE :
   - Seuls les vrais messages CEO ↔ agent sont sauvegardés en DB
   - Les prompts internes (briefs de délégation, synthèse) ne polluent pas la mémoire
   - Filtre au chargement pour nettoyer d'éventuels messages internes anciens

   AUTRES FEATURES CHAT :
   - Joindre des fichiers (texte : contenu injecté dans le message, image : Claude Vision)
   - Bouton téléchargement .md sur les messages avec du code (nomé agent-livrable-date.md)
   - Mémoire persistante (10 derniers échanges utilisés comme contexte Claude)
   - Markdown rendu avec react-markdown + remark-gfm

4. PROJETS MULTI-AGENTS (app/project/page.tsx)
   Architecture en 2 étapes pour éviter les timeouts Netlify :
   Étape 1 : POST /api/project → crée le projet en DB, retourne l'ID immédiatement
   Étape 2 : POST /api/project/[id]/run → lance l'orchestration (fire-and-forget)

   Le client poll /api/project/[id]/logs toutes les 800ms
   et affiche les logs en temps réel (typés : system / thinking / output / success / error).

   Flow d'orchestration :
   - Claude décompose le brief en tâches par agent (3 à 6 agents)
   - Chaque agent traite sa tâche dans l'ordre
   - Les logs affichent "Nom — Rôle" pour chaque agent
   - Timeout client de 3 minutes avec AbortController
   - Si timeout → statut automatiquement mis à "paused" via PATCH /api/project


APPELS CLAUDE API (lib/claude.ts)
----------------------------------
Chaque appel suit ce pattern :
1. Récupérer le profil de l'agent depuis Supabase
2. Récupérer les 10 derniers messages depuis memories
3. Construire le system prompt : personality + knowledge SURGIFLOW + règles de délégation
4. Appeler claude-sonnet-4-20250514
5. Parser les blocs <DELEGATE>...</DELEGATE> (N blocs possibles pour N délégations)
6. Sauvegarder user message + réponse en memories (sauf si options.saveUserMessage = false)
7. Retourner la réponse + delegations[]

Fonctions principales :
- callAgent(agentId, message, projectId?, extraContext?, options?)
    → options.saveUserMessage : false pour les briefs internes (évite pollution mémoire)
    → options.maxTokens : override du max_tokens (1024 pour briefs, 2048 par défaut)
- callAgentWithImage() : appel avec image base64 (Claude Vision)
- callClaudeRaw() : appel sans persistance (orchestration interne)
- orchestrateProject() : demande à Claude de décomposer un brief en tâches JSON
- generateRecruitmentRequests() : génère les demandes de recrutement d'un C-Level
- parseDelegations() : extrait tous les blocs <DELEGATE> d'une réponse (regex global)

Format de délégation dans les réponses Claude :
<DELEGATE>{"agentName": "Léa", "agentRole": "Développeuse Frontend", "task": "..."}</DELEGATE>
<DELEGATE>{"agentName": "Samy", "agentRole": "DevOps", "task": "..."}</DELEGATE>
(autant de blocs que nécessaire)


STRUCTURE DES FICHIERS
----------------------

  api/
    agents/route.ts              → GET liste tous les agents
    agents/[id]/route.ts         → GET un agent par ID

    synthesize/route.ts          → POST synthèse du manager (guide d'implémentation CEO)

    recruit/route.ts             → GET/POST/PATCH recrutement
  dashboard/page.tsx             → vue équipe organigramme
  recruit/page.tsx               → recrutement + validation (matching par nom)
  project/page.tsx               → lancement projet + logs temps réel
  chat/[agentId]/page.tsx        → chat avec délégation autonome en temps réel
  layout.tsx                     → layout global (h-screen, sidebar sticky)

  Sidebar.tsx                    → navigation sticky (md: h-screen sticky top-0, mobile: drawer)

  PageHeader.tsx                 → header de page responsive

lib/

  claude.ts                      → wrapper Claude API complet + parseDelegations

supabase/
  migrations/001_initial_schema.sql → schéma SQL + RLS
  seed.ts                            → insertion des 13 agents avec hiérarchie manager_id


GESTION DES TIMEOUTS NETLIFY
-----------------------------
Netlify coupe les connexions inactives après ~26 secondes.
Stratégies utilisées :
- Briefs des agents par paquets de 2 (pas tout en parallèle)
- max_tokens réduit à 1024 pour les briefs (réponses 2x plus rapides)
- Retry automatique (1 tentative) avec délai de 2s si erreur
- export const maxDuration = 300 sur les routes longues
- Projet : fire-and-forget + polling côté client


SÉCURITÉ
--------
- Toutes les clés API dans les variables d'environnement
- RLS (Row Level Security) activé sur toutes les tables Supabase
- Aucune clé exposée côté client
- Tous les appels Claude côté serveur (Next.js API routes)
- Prompts internes non sauvegardés en mémoire (pas de pollution CEO)

DÉPLOIEMENT (Netlify)
----------------------
1. Créer un projet sur supabase.com
2. Exécuter supabase/migrations/001_initial_schema.sql dans le SQL Editor
3. Remplir .env.local et lancer : npm run seed
4. Importer le repo sur netlify.com, renseigner les 4 variables d'env dans Site Settings
5. netlify.toml configure le build (command: npm run build, publish: .next, NODE_VERSION: 20)
6. @netlify/plugin-nextjs gère les API routes comme des Netlify Functions


BUGS CONNUS ET RÉSOLUS
-----------------------
- Inès apparaissait "non recrutée" → matching par nom au lieu de rôle (rôles différents entre CLEVELS et seed)
- Sidebar disparaissait au scroll → body h-screen + sidebar sticky top-0 h-screen
- Messages internes sauvegardés comme messages CEO → option saveUserMessage: false
- Tout perdre au refresh pendant délégation → localStorage + beforeunload warning
- 3/4 agents timeout en parallèle → paquets de 2 + retry + tokens réduits
- Cartes de délégation invisibles (cachées dans la bulle Marcus) → messages "delegation" séparés dans le fil
